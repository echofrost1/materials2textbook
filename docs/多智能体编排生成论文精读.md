## 1. 建议你们项目采用的总流程

### 1.1 从论文抽象出的系统流程

```text
用户输入教材主题与资料范围
        ↓
Resource Collector Agent
收集网页、PDF、教材、课件、视频等候选资料
        ↓
Parser Agent
解析文本、页面、图片、表格、公式、字幕、关键帧
        ↓
Retrieval Builder Agent
构建文本向量索引、多模态索引、知识图谱和层次摘要树
        ↓
Relevance Judge Agent
按主题相关性、信息密度、可教学性、来源可靠性过滤资料
        ↓
Knowledge Organizer Agent
生成知识点树、章节结构、概念关系、学习路径
        ↓
Textbook Writer Agent
基于证据片段写章节正文、例子、习题、思考题
        ↓
Evidence Verifier Agent
检查引用、事实一致性、证据覆盖率和幻觉风险
        ↓
Pedagogy Reviewer Agent
检查教学目标、难度梯度、讲解逻辑、学生可理解性
        ↓
Revision Agent
根据审核意见修订内容
        ↓
输出 Markdown / HTML / DOCX 数字教材
```

### 1.2 推荐的 Agent 分工

| Agent | 来自哪些论文的启发 | 核心职责 |
|---|---|---|
| Resource Collector | Agentic AutoSurvey, AutoGen | 检索与收集候选资料 |
| Parser Agent | RAG-Anything, VisRAG, VideoRAG | 多源资料解析与结构化 |
| Retrieval Builder | GraphRAG, RAPTOR, RAG-Anything | 建索引、图谱、层次摘要 |
| Relevance Judge | CRAG, Self-RAG | 判断资料是否可靠、有用 |
| Knowledge Organizer | GraphRAG, RAPTOR, Agentic AutoSurvey | 聚类知识点，生成章节树 |
| Textbook Writer | AI-Augmented Textbook, Interactive Lessons | 生成教材正文与学习活动 |
| Evidence Verifier | Self-RAG, CRAG, Why Do MAS Fail | 检查引用、事实和证据支撑 |
| Pedagogy Reviewer | LLM Agents for Education | 检查教学逻辑和学习目标 |
| Revision Agent | MetaGPT, AutoGen | 接收反馈并修订输出 |

### 1.3 可以写进项目报告的核心表述

本项目不采用单模型一步生成教材，而是参考多智能体编排与 Agent workflow 思路，将数字教材生成拆解为资料收集、资料解析、知识组织、内容写作、证据审核、教学审查和修订输出等多个可追踪阶段。不同 Agent 负责不同中间产物，并通过标准化流程传递结果，从而降低单次生成的幻觉风险，提高教材内容的可控性、可解释性和可评估性。

## 2. 第一组：多智能体编排生成主线

### 2.1 AutoGen

PDF：`01_AutoGen_Enabling_Next_Gen_LLM_Applications_via_Multi_Agent_Conversation.pdf`

**核心问题**

复杂 LLM 应用不是单次问答，而是需要多个 Agent、工具、人类输入和不同执行模式协作完成。AutoGen 的重点是把这种协作统一成“可对话 Agent + 可编程对话流程”。

**论文流程**

```text
定义 conversable agents
        ↓
给每个 Agent 分配角色、能力和工具
        ↓
通过 conversation programming 定义交互规则
        ↓
Agent 之间交换消息、调用工具、请求人类反馈
        ↓
完成数学、代码、问答、决策等复杂任务
```

**精华**

- Agent 是可对话实体，不只是 prompt 模板。
- Agent 可以组合 LLM、工具、人类输入和代码执行。
- 编排的关键不是“Agent 多”，而是“谁和谁通信、何时通信、通信后如何决策”。
- 对话流程可以被编程，这一点适合你们把教材生成流程固定下来。

**对你们项目的启发**

- 把教材生成定义为多 Agent 会话流程。
- Resource Collector、Writer、Verifier、Reviewer 可以通过消息交互协作。
- 教师可以作为 Human Agent 插入审核节点。
- 每个 Agent 的输入输出都要结构化，避免纯自然语言传递导致信息丢失。

**可写进报告**

参考 AutoGen 的多智能体对话框架，本项目将数字教材生成建模为多个可对话 Agent 的协作过程，通过角色定义、消息传递和工具调用实现资料检索、内容生成与质量审核的流程化编排。

### 2.2 MetaGPT

PDF：`02_MetaGPT_Meta_Programming_for_A_Multi_Agent_Collaborative_Framework.pdf`

**核心问题**

普通多 Agent 系统容易因为盲目串联 LLM 而产生级联幻觉和逻辑不一致。MetaGPT 的思路是把人类工作流程中的 SOP 编码进多 Agent 系统。

**论文流程**

```text
用户提出复杂任务
        ↓
系统将任务映射到标准化 SOP
        ↓
不同角色 Agent 按流水线处理
        ↓
每个阶段生成明确中间产物
        ↓
后续 Agent 基于前一阶段产物继续加工和校验
        ↓
输出更一致的完整方案
```

**精华**

- 多智能体不是自由聊天，而是 SOP 驱动的协作。
- 中间产物非常重要，可以降低信息丢失和幻觉累积。
- 角色分工应该贴近真实人类工作流程。
- 流水线式 Agent 协作适合复杂生成任务。

**对你们项目的启发**

你们可以直接设计一套“教材生成 SOP”：

```text
确定教材主题
→ 搜索资料
→ 解析资料
→ 过滤资料
→ 聚类知识点
→ 生成章节目录
→ 写章节正文
→ 插入媒体素材
→ 审核事实和引用
→ 教学审查
→ 修订输出
```

**可写进报告**

参考 MetaGPT 的 SOP 驱动思想，本项目将教材生成任务拆解为一系列标准化阶段，并要求每个 Agent 输出可复用的中间结果，如知识点树、证据片段表、章节草稿和审核意见，从而提升生成过程的稳定性。

### 2.3 Agentic AutoSurvey

PDF：`03_Agentic_AutoSurvey_Let_LLMs_Survey_LLMs.pdf`

**核心问题**

科学文献增长太快，人工综述成本高。Agentic AutoSurvey 用多个专门 Agent 自动完成论文检索、主题聚类、综述写作和质量评估。

**论文流程**

```text
输入综述主题
        ↓
Paper Search Specialist 检索论文
        ↓
Topic Mining & Clustering Agent 挖掘主题并聚类
        ↓
Academic Survey Writer Agent 写综述
        ↓
Quality Evaluator Agent 从多个维度评价质量
        ↓
输出结构化学术综述
```

**精华**

- 它和你们项目最像，只是输出从“综述”换成“教材”。
- 论文强调专业 Agent 分工比单模型直接写更好。
- 评估维度不仅看是否流畅，还看组织性、综合能力、批判性分析和引用覆盖。
- 可以处理大量论文，说明多 Agent 对资料密集型生成任务有优势。

**对你们项目的启发**

可直接迁移为：

```text
Resource Search Agent
→ Topic Mining & Clustering Agent
→ Textbook Writer Agent
→ Quality Evaluator Agent
```

进一步扩展：

- Paper Search 变成多源资料搜索。
- Topic Mining 变成知识点聚类和章节规划。
- Survey Writer 变成教材写作。
- Quality Evaluator 变成事实审核、教学审核、引用审核。

**可写进报告**

Agentic AutoSurvey 证明了多智能体架构适用于资料密集型长文本生成任务。本项目借鉴其“检索-聚类-写作-评估”流程，将其迁移到数字教材生成场景。

### 2.4 A Survey on Agent Workflow

PDF：`04_A_Survey_on_Agent_Workflow_Status_and_Future.pdf`

**核心问题**

单个 Agent 难以应对复杂、多步骤、动态任务，因此需要 Agent workflow 作为编排基础设施。

**论文流程抽象**

```text
任务目标
        ↓
规划节点
        ↓
工具调用节点
        ↓
记忆与状态管理
        ↓
多 Agent 协作节点
        ↓
条件判断与循环修订
        ↓
输出与评估
```

**精华**

- Workflow 是管理任务顺序、依赖和状态的系统。
- Agent workflow 常用图结构表示，节点是处理步骤，边是依赖关系。
- 真实系统需要规划、记忆、工具调用、多 Agent 协作、安全控制和优化。
- 多模态集成和标准化是未来重点。

**对你们项目的启发**

你们的系统最好用 LangGraph 这类图式工作流实现，而不是简单写几个连续 prompt。每个节点可对应一个 Agent 或工具：

- 节点：解析 PDF
- 节点：视频 ASR
- 节点：知识点抽取
- 节点：章节规划
- 节点：写作
- 节点：审核
- 条件边：审核不通过则回到修订节点

**可写进报告**

本项目将多智能体系统设计为图式 Agent workflow，通过节点和边显式表示教材生成中的任务阶段、依赖关系和审核修订循环，从而增强系统的可控性和可解释性。

### 2.5 Multi-Agent Collaboration Mechanisms

PDF：`05_Multi_Agent_Collaboration_Mechanisms_A_Survey_of_LLMs.pdf`

**核心问题**

多智能体系统需要回答“谁参与、怎么协作、由谁协调、采用什么策略和通信协议”等问题。

**论文分类框架**

```text
Actors: 哪些 Agent 参与
Types: 合作、竞争、合作竞争
Structures: 点对点、中心化、分布式
Strategies: 角色驱动、模型驱动、能力驱动
Protocols: 通信协议、协调规则、任务分配机制
```

**精华**

- 协作结构会影响系统稳定性。
- 中心化编排适合流程明确的工程任务。
- 去中心化协作适合开放讨论和多观点探索。
- 对抗式或竞争式 Agent 可以用于审核和挑错。

**对你们项目的启发**

建议采用“中心化流程控制 + 局部对抗审核”：

- 中心化控制：由 Workflow Manager 控制任务流。
- 串行流水线：资料解析、筛选、组织、写作依次进行。
- 并行专家：多个 Reviewer 同时检查事实、教学、引用。
- 对抗审核：Verifier 专门找 Writer 的错误。

**可写进报告**

本项目采用中心化多智能体编排结构，由工作流管理器控制任务推进，同时引入审核类 Agent 对生成内容进行事实、引用和教学逻辑检查，形成生成与评估相结合的协作机制。

## 3. 第二组：教育内容生成与质量控制

### 3.1 Towards an AI-Augmented Textbook

PDF：`06_Towards_an_AI_Augmented_Textbook.pdf`

**核心问题**

传统教材是一种固定媒介，难以适配不同学习者。该论文提出用生成式 AI 对教材进行转换和增强，形成个性化、多表示的学习体验。

**论文流程**

```text
输入传统教材章节
        ↓
保持原始内容完整性
        ↓
生成多种学习表达
        ↓
加入个性化学习支持
        ↓
通过教学专家评估和学习实验验证效果
```

**精华**

- 教材生成不是简单摘要，而是学习内容重构。
- 生成内容必须保持原始知识完整性。
- 可以把同一知识转成文本、幻灯片、音频、视频、思维导图、测验等多种形式。
- 需要教育评价，而不只是语言质量评价。

**对你们项目的启发**

你们的数字教材输出可以不止正文，还可以包含：

- 章节导读
- 核心概念
- 多媒体素材
- 例题
- 思考题
- 学习路径
- 参考链接

**可写进报告**

参考 AI-Augmented Textbook 的内容增强思想，本项目将多源资料重构为面向学习者的数字教材，不仅生成章节正文，还生成知识点、案例、媒体素材、练习题和引用来源。

### 3.2 Automatic LLM Creation of Interactive Learning Lessons

PDF：`07_Automatic_LLM_Creation_of_Interactive_Learning_Lessons.pdf`

**核心问题**

LLM 能否自动生成结构化互动课程？论文比较了单步生成和任务分解式生成，结果显示任务分解策略效果更好。

**论文流程**

```text
输入课程主题和教学场景
        ↓
RAG 检索相关教学资料
        ↓
将课程生成拆成多个子任务
        ↓
分别生成目标、场景、互动环节、反馈、总结
        ↓
人工用 rubric 评价生成质量
```

**精华**

- 一步生成课程质量不稳定。
- 任务分解式 prompting 更适合复杂教学内容生成。
- 教学内容评价需要 rubric，不能只看文本流畅度。
- 人机混合更现实：AI 初稿，人工审核和完善。

**对你们项目的启发**

你们的教材生成也应该拆分：

```text
章节目标生成
→ 知识点解释
→ 案例生成
→ 媒体素材匹配
→ 习题生成
→ 章节总结
→ 审核修订
```

**可写进报告**

参考互动课程自动生成研究，本项目避免单次 prompt 直接生成完整教材，而采用任务分解式生成策略，将章节正文、案例、习题和总结分阶段生成。

### 3.3 LLM Agents for Education

### 3.4 Self-RAG

PDF：`09_Self_RAG_Learning_to_Retrieve_Generate_and_Critique.pdf`

**核心问题**

普通 RAG 常常固定检索若干文档，不判断是否真的需要检索，也不判断检索内容是否支持生成结果。Self-RAG 提出检索、生成和自我批判结合。

**论文流程**

```text
判断是否需要检索
        ↓
检索相关证据
        ↓
基于证据生成回答
        ↓
判断证据是否相关
        ↓
判断生成内容是否被证据支持
        ↓
必要时修正输出
```

**精华**

- 检索不是每次都必须做，应按需触发。
- 生成后要判断内容是否有证据支撑。
- 审核可以成为生成过程的一部分，而不是事后人工看一眼。
- 对长文本生成的事实性和引用准确性有帮助。

**对你们项目的启发**

Evidence Verifier Agent 可以执行：

```json
{
  "claim": "教材中的某个事实陈述",
  "need_retrieval": true,
  "evidence_relevant": true,
  "claim_supported": true,
  "revision_required": false
}
```

**可写进报告**

本项目借鉴 Self-RAG 的自反思思想，在教材生成后引入证据相关性与事实支撑性检查，要求关键结论必须能够回溯到原始资料片段。

### 3.5 Corrective RAG

PDF：`10_Corrective_Retrieval_Augmented_Generation.pdf`

**核心问题**

RAG 严重依赖检索质量。如果检索结果错误，模型可能被错误资料误导。CRAG 提出先评估检索质量，再决定采用、补充或纠正。

**论文流程**

```text
输入问题
        ↓
检索候选文档
        ↓
Retrieval Evaluator 判断文档质量
        ↓
若文档可靠：直接使用
若文档模糊：补充搜索
若文档错误：触发纠正策略
        ↓
对文档分解和重组，提取关键知识片段
        ↓
生成答案
```

**精华**

- 检索结果要先评估，不能无条件塞给模型。
- 可设计 Correct / Ambiguous / Incorrect 三类动作。
- 长文档应分解成更细粒度的知识片段，再选择关键部分。
- 适合作为资料过滤和证据审核模块。

**对你们项目的启发**

资料筛选 Agent 可以输出：

```json
{
  "retrieval_quality": "ambiguous",
  "action": "web_search_extend",
  "reason": "候选资料相关但来源不够权威，需要补充资料验证"
}
```

**可写进报告**

参考 CRAG 的纠错式检索思想，本项目在资料进入教材写作前增加检索质量评估，对低质量或不确定资料触发补充检索和证据重筛。

## 4. 第三组：多源资料处理与知识组织底座

### 4.1 RAG-Anything

PDF：`11_RAG_Anything_All_in_One_RAG_Framework.pdf`

**核心问题**

真实资料不是纯文本，而是包含文本、图像、表格、公式等多种模态。传统文本 RAG 会丢失跨模态结构信息。

**论文流程**

```text
输入多模态文档
        ↓
识别文本、图片、表格、公式等元素
        ↓
将不同模态视为知识实体
        ↓
构建跨模态关系图和文本语义图
        ↓
进行结构导航 + 语义匹配的混合检索
        ↓
为生成提供跨模态证据
```

**精华**

- 多模态内容不应强行转成纯文本。
- 文本、图、表、公式可以被建模为互相关联的知识实体。
- 双图结构能同时保留跨模态关系和文本语义关系。
- 对长文档和复杂版面更有优势。

**对你们项目的启发**

教材资源库中每个资料片段可以建成实体：

- 文本段落
- 图片
- 表格
- 公式
- 视频片段
- 关键帧
- 源链接

实体之间建立关系：

- 图片解释某概念
- 表格支持某结论
- 视频片段讲解某知识点
- 段落引用某公式

**可写进报告**

本项目借鉴 RAG-Anything 的跨模态知识实体建模思想，将文本、图像、表格、公式和视频片段统一表示为可检索、可关联的资源实体，为数字教材生成提供多模态证据支持。

---

### 4.2 GraphRAG

PDF：`12_GraphRAG_From_Local_to_Global.pdf`

**核心问题**

普通 RAG 擅长回答局部事实问题，但不擅长回答全局问题，如“整个资料库有哪些主题”。GraphRAG 用知识图谱和社区摘要来支持全局理解。

**论文流程**

```text
输入大规模文本语料
        ↓
LLM 抽取实体和关系
        ↓
构建实体知识图谱
        ↓
图聚类得到社区
        ↓
为每个社区生成摘要
        ↓
查询时先生成多个社区级回答
        ↓
再汇总成最终回答
```

**精华**

- 章节规划属于全局理解任务，不是简单 chunk 检索。
- 知识图谱可以帮助发现主题、概念和关系。
- 社区摘要适合从海量资料中提炼教材章节。
- 对“生成目录、发现主题、组织知识点”特别有用。

**对你们项目的启发**

Knowledge Organizer Agent 可以这样做：

```text
抽取概念与关系
→ 构建知识图谱
→ 发现主题社区
→ 将社区映射为教材章节
→ 为每章生成摘要和知识点列表
```

**可写进报告**

参考 GraphRAG 的全局知识组织方法，本项目通过实体关系抽取和社区摘要从多源资料中发现主题结构，并据此自动生成数字教材目录和章节知识点。

---

### 4.3 RAPTOR

PDF：`13_RAPTOR_Recursive_Abstractive_Processing_for_Tree_Organized_Retrieval.pdf`

**核心问题**

普通 RAG 只检索短 chunk，缺乏对长文档整体层次结构的理解。RAPTOR 通过递归聚类和摘要构建树状知识结构。

**论文流程**

```text
原始文档切分为 chunks
        ↓
对 chunks 做 embedding
        ↓
聚类相似 chunks
        ↓
为每个聚类生成摘要
        ↓
摘要继续聚类和摘要
        ↓
形成多层知识树
        ↓
检索时同时利用底层细节和高层摘要
```

**精华**

- 层次化摘要适合长文档。
- 树结构可以同时支持细节检索和全局理解。
- soft clustering 允许同一片段属于多个主题。
- 非常适合教材章节树生成。

**对你们项目的启发**

可以把资源组织为：

```text
资料片段层
→ 知识点层
→ 小节层
→ 章节层
→ 教材总目录层
```

**可写进报告**

本项目借鉴 RAPTOR 的递归摘要与树状检索机制，将多源资料从片段级逐步聚合为知识点、小节和章节，从而支撑教材目录与正文生成。

---

### 4.4 Multimodal RAG Surveys

PDF：

- `14_Ask_in_Any_Modality_Comprehensive_Survey_on_Multimodal_RAG.pdf`
- `15_A_Survey_of_Multimodal_Retrieval_Augmented_Generation.pdf`

**核心问题**

多模态 RAG 需要处理文本、图像、音频、视频等多种信息，关键挑战是跨模态检索、融合、增强和生成。

**通用流程**

```text
多模态数据输入
        ↓
模态解析与表示学习
        ↓
跨模态检索
        ↓
多模态证据融合
        ↓
生成回答或内容
        ↓
评估事实性、相关性、跨模态一致性
```

**精华**

- 多模态 RAG 不只是“图片转文字后再 RAG”。
- 跨模态对齐是核心难点。
- 检索、融合、生成、评估都需要适配多模态。
- Agent-based MRAG 是未来方向之一。

**对你们项目的启发**

数字教材生成中，系统要能回答：

- 这个图片适合放在哪一章？
- 这个视频片段支撑哪个知识点？
- 这张表格能否作为某段结论的证据？
- 生成正文是否正确引用了图片、视频、表格？

**可写进报告**

本项目将多模态 RAG 作为多源资源聚合的底层能力，通过跨模态检索和证据融合，将文本、图像、表格、音频和视频资源统一服务于教材生成。

### 4.5 VisRAG

PDF：`16_VisRAG_Vision_based_Retrieval_augmented_Generation.pdf`

**核心问题**

传统文档 RAG 先 OCR 或解析成文本，会丢失页面布局、图像、表格和视觉线索。VisRAG 直接把文档页面作为图像进行检索和生成。

**论文流程**

```text
输入 PDF / 文档页面图像
        ↓
VLM-based Retriever 直接编码页面图像
        ↓
根据查询检索相关页面
        ↓
VLM-based Generator 基于页面图像回答
        ↓
保留布局、图像和视觉结构信息
```

**精华**

- 对教材、PPT、扫描件，视觉布局很重要。
- 先转纯文本可能损失关键信息。
- 页面级视觉检索适合复杂文档。
- 可以和文本 RAG 互补。

**对你们项目的启发**

处理 PDF / PPT 时可以双通道：

- 文本通道：OCR、正文、标题、公式。
- 视觉通道：页面截图、图表、布局。

然后由 Media Selector Agent 判断哪些图表适合插入教材。

**可写进报告**

参考 VisRAG，本项目在处理 PDF、PPT 和扫描教材时保留页面视觉信息，避免仅依赖 OCR 文本导致图表、布局和公式信息丢失。

## 5. 第四组：长视频理解

### 5.1 VideoRAG

PDF：`17_VideoRAG_Extreme_Long_Context_Videos.pdf`

**核心问题**

长视频包含大量视觉和语义信息，普通文本 RAG 难以处理。VideoRAG 提出双通道架构：图文本知识 grounding + 多模态上下文编码。

**论文流程**

```text
输入长视频集合
        ↓
切分视频片段
        ↓
提取字幕、视觉特征、关键帧
        ↓
构建跨视频知识图谱
        ↓
建立多模态检索索引
        ↓
根据查询检索相关视频片段
        ↓
结合文本和视觉信息生成回答
```

**精华**

- 长视频需要片段级索引。
- 视频理解不能只靠字幕，也要保留视觉信息。
- 跨视频知识图谱可以支持多视频资料组织。
- 很适合“几十小时视频中找教学片段”。

**对你们项目的启发**

视频资源可以结构化为：

```json
{
  "clip_id": "video_001_00:12:30_00:14:05",
  "transcript": "字幕文本",
  "keyframes": ["frame1.jpg", "frame2.jpg"],
  "knowledge_points": ["概念A", "方法B"],
  "chapter_candidate": "第二章",
  "evidence_score": 0.86
}
```

**可写进报告**

本项目借鉴 VideoRAG 的长视频处理思路，将课程视频切分为片段级资源，并结合字幕、关键帧和语义标签构建可检索的视频知识库。

### 5.2 Video Understanding with Large Language Models

PDF：`18_Video_Understanding_with_Large_Language_Models_A_Survey.pdf`

**核心问题**

视频理解任务越来越复杂，LLM 可以作为总结器、管理器、文本解码器、回归器或隐藏层参与视频理解。

**论文分类**

```text
Video Analyzer × LLM
Video Embedder × LLM
(Analyzer + Embedder) × LLM
```

LLM 的角色包括：

- Summarizer
- Manager
- Text Decoder
- Regressor
- Hidden Layer

**精华**

- 视频理解不仅是分类，还包括摘要、问答、时间推理和事件理解。
- 长视频需要多粒度理解：整体、时间段、局部事件。
- LLM 可以作为管理器协调多个视频工具。

**对你们项目的启发**

视频处理 Agent 可以分成：

- ASR Agent：抽字幕。
- Keyframe Agent：抽关键帧。
- Video Summary Agent：生成片段摘要。
- Temporal Locator Agent：定位知识点出现时间。
- Media Selector Agent：选择可插入教材的片段。

**可写进报告**

参考视频大模型综述，本项目将长视频理解拆解为字幕转写、关键帧抽取、片段摘要、时间定位和知识点标注等子任务，供多智能体工作流调用。

### 5.3 Video-ChatGPT

PDF：`19_Video_ChatGPT_Towards_Detailed_Video_Understanding.pdf`

**核心问题**

视频模型需要支持开放式对话，而不仅是固定任务分类。Video-ChatGPT 将视频视觉编码器和 LLM 连接起来，并用视频指令数据训练。

**论文流程**

```text
收集视频
        ↓
构建视频指令数据
        ↓
视频编码器提取时空特征
        ↓
通过 adapter 接入 LLM
        ↓
训练模型进行视频对话
        ↓
用视频问答和开放描述评价
```

**精华**

- 视频对话可以成为统一接口。
- 视频指令数据对视频理解能力很重要。
- 对长视频和细微时间关系仍有困难。

**对你们项目的启发**

你们不一定要训练 Video-ChatGPT，但可以借鉴接口形态：

```text
用户问：这个视频里哪一段适合讲“图卷积”？
系统答：00:12:30-00:14:05，原因是该片段解释了邻接矩阵和消息传递机制。
```

**可写进报告**

参考 Video-ChatGPT，本项目将视频资源转化为可问答、可摘要、可定位的教学资源，使长视频能够被教材生成流程有效调用。

## 6. 第五组：多智能体失败模式与评估

### 6.1 Why Do Multi-Agent LLM Systems Fail?

PDF：`20_Why_Do_Multi_Agent_LLM_Systems_Fail.pdf`

**核心问题**

多 Agent 不一定比单 Agent 好。论文系统分析多智能体系统失败原因，提出失败分类。

**论文流程**

```text
收集多个 MAS 框架的运行轨迹
        ↓
人工分析失败案例
        ↓
构建失败分类体系 MAST
        ↓
用 LLM-as-a-Judge 扩展标注
        ↓
分析不同模型、任务和框架的失败模式
```

**失败类型**

论文将失败归为三大类：

- 系统设计问题
- Agent 间目标或信息不一致
- 任务验证不足

**精华**

- 多 Agent 的失败常常不是模型能力问题，而是系统设计问题。
- Agent 角色不清会导致重复劳动或互相推诿。
- 信息传递丢失会导致后续 Agent 基于错误上下文工作。
- 审核 Agent 可能只是表面审核，不能真正发现错误。

**对你们项目的启发**

需要提前设计防失败机制：

- 每个 Agent 有明确输入输出 schema。
- 每个阶段保存中间产物。
- 审核 Agent 必须给出证据和错误位置。
- 任务要有终止条件，避免无限修订。
- 不通过审核时只回退到相关阶段，而不是全流程重跑。

**可写进报告**

考虑到多智能体系统可能存在角色不清、信息传递丢失和任务验证不足等失败模式，本项目通过标准化中间表示、显式审核节点和修订终止条件提升系统可靠性。

### 6.2 Evaluation and Benchmarking of LLM Agents

PDF：`21_Evaluation_and_Benchmarking_of_LLM_Agents_A_Survey.pdf`

**核心问题**

LLM Agent 的评价还不成熟。论文提出从“评价什么”和“如何评价”两个维度组织 Agent 评估。

**评价框架**

```text
评价什么：
能力、行为、可靠性、安全性

如何评价：
交互模式、数据集、指标、工具、人工或自动评估
```

**精华**

- Agent 评价不能只看最终答案。
- 需要评价执行过程、工具使用、行为稳定性和安全性。
- 长程、多轮、真实环境中的评价尤其重要。
- 自动化评价和人类评价需要结合。

**对你们项目的启发**

建议评价指标：

| 维度 | 指标 |
|---|---|
| 检索能力 | 找到相关资料的比例 |
| 资料质量 | 来源可靠性、版权可用性 |
| 组织能力 | 章节结构是否合理 |
| 生成质量 | 内容完整性、语言清晰度 |
| 事实可靠性 | 引用准确率、证据覆盖率 |
| 教学质量 | 难度梯度、学习目标、例题质量 |
| 流程稳定性 | Agent 是否按流程工作 |
| 人工节省 | 相比人工整理减少多少时间 |

**可写进报告**

本项目从结果质量和过程行为两个层面评估多智能体教材生成系统，既评价最终教材的完整性、事实性和教学性，也评价 Agent 工作流的执行稳定性和中间产物质量。

## 8. 建议的 MVP 实现路径

### 第一版：先做文本和 PDF

1. 输入一个教材主题和若干 PDF / 网页。
2. Parser Agent 解析文本并分块。
3. Retrieval Builder 构建向量库。
4. Knowledge Organizer 生成知识点树和目录。
5. Textbook Writer 生成章节正文。
6. Evidence Verifier 检查每段是否有证据。
7. 输出 Markdown 教材。

### 第二版：加入多 Agent 审核

1. Relevance Judge 过滤资料。
2. Evidence Verifier 检查引用。
3. Pedagogy Reviewer 检查教学逻辑。
4. Revision Agent 根据审核意见修订。

### 第三版：加入多模态和视频

1. PDF 页面图像和图表入库。
2. 视频切片、ASR、关键帧入库。
3. Media Selector 自动选择图片和视频片段。
4. 输出包含媒体引用的 HTML 数字教材。

## 9. 推荐汇报时的讲法

### 9.1 不建议这么讲

我们做了一个多模态 RAG 系统，可以从网页、PDF、视频里找资料，然后生成教材。

这个说法会让重点偏向 RAG，而不是老师要求的多智能体编排生成。

### 9.2 建议这么讲

我们关注的是面向数字教材生成的多智能体编排方法。系统将教材生成拆分为资料收集、资料解析、知识组织、教材写作、证据审核、教学审查和修订输出等阶段，并为每个阶段设计专门 Agent。多模态 RAG、GraphRAG、RAPTOR 和视频理解作为资料处理底座，为多智能体生成流程提供可检索、可引用、可追踪的证据支持。

## 10. 一句话精华

> 这些论文共同说明：高质量教材生成不能依赖单个 LLM 一步完成，而应通过多智能体工作流把检索、组织、写作、审核和修订拆成可控阶段，并用多模态 RAG 与知识图谱提供证据底座。
