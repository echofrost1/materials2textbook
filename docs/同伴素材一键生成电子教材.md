# 同伴素材一键生成电子教材

适用场景：素材、台账、切片、PPT 证据和转码结果都在同伴电脑本地的 `work_materials/work_material1/` 目录中；同伴拉取本仓库更新后，希望直接跑出可打开的电子教材。

## 目录要求

脚本默认读取：

```text
work_materials/work_material1/
├── 01_manifest_inventory/
│   ├── assets_manifest.xlsx
│   ├── active_assets.xlsx
│   ├── asset_block_map.xlsx
│   ├── chapter_asset_map_reviewed.xlsx
│   ├── ppt_assets.xlsx
│   └── video_segments.xlsx
└── 02_working_processing/
    ├── converted_mp4/
    ├── keyframes/
    ├── ppt_images/
    └── json/
        ├── video_segments.jsonl
        └── ppt_assets.jsonl
```

其中 `asset_block_map.xlsx` 是当前整本教材规划的优先依据。程序会读取其中的 `material_block_cn` 作为章、`knowledge_point_cn` 作为节/知识点，并结合内置焊接课程排序规则自动生成稳定的全书顺序；缺失或未命中的素材会按标题、关键词、已有章节字段自动补齐，不能补齐的内容进入引用池或后置章节。

`02_working_processing/json/` 下的 JSONL 仍是正文生成、视频块、PPT/文档证据和问书检索的主要输入。xlsx 负责“素材属于哪章哪节”，JSONL 负责“教材正文能引用什么证据”。

## 一键生成

在仓库根目录运行：

```powershell
pip install -r requirements.txt
python scripts/run_full_digital_textbook.py `
  --book-mode `
  --manifest-xlsx work_materials/work_material1/01_manifest_inventory/asset_block_map.xlsx `
  --material-root work_materials/work_material1 `
  --title "焊接技术数字教材" `
  --max-video-records 40 `
  --max-document-records 500 `
  --output-dir work_materials/work_material1/05_final_deliverables/agent_workflow `
  --student-package-output work_materials/work_material1/05_final_deliverables/digital_book.zip
```

默认不会复制视频大文件，而是在 `digital_book.json` 中用相对路径引用 `02_working_processing/converted_mp4/` 和 `02_working_processing/keyframes/`。这适合几个 G 到上百 G 的素材目录，避免每次生成教材都重复复制媒体资源。

生成结果：

```text
work_materials/work_material1/05_final_deliverables/
├── agent_workflow/
│   ├── book_plan.json
│   ├── book_outline.md
│   ├── book_plan_review.md/json
│   ├── curriculum_order.generated.yml
│   ├── textbook_outline.md/json
│   ├── evidence_chunks.jsonl
│   ├── chapter_plan.json
│   ├── textbook_draft.md/docx
│   ├── review_report.md/json
│   ├── textbook_final.md/docx
│   ├── workflow_summary.json
│   └── artifact_manifest.json
└── digital_book/
    ├── digital_book.json
    ├── index.html
    ├── styles.css
    ├── app.js
    ├── ask_config.js
    └── assets/
```

打开电子教材：

```powershell
python scripts/open_digital_book.py
```

或者双击仓库根目录下的：

```text
open_digital_book.bat
```

脚本会自动把 HTTP 服务根目录指向 `work_materials/work_material1`，再打开：

```text
http://127.0.0.1:8767/05_final_deliverables/digital_book/index.html
```

这样视频、关键帧和教材 JSON 的相对路径都能正常访问。若 8767 端口被占用，脚本会自动换一个可用端口。

如果需要生成一个可拷贝给别人、媒体资源也打包进 `digital_book/assets/` 的小样例包，可以显式复制媒体：

```powershell
python scripts/run_full_digital_textbook.py --material-root work_materials/work_material1 --copy-media-assets
```

注意：素材达到几十 G 或上百 G 时，不建议使用 `--copy-media-assets`。

## 分章/分节生成

大教材建议按章或按知识点分批生成，避免一次处理全量证据和全量媒体：

```powershell
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作"
```

只生成某个知识点：

```powershell
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --knowledge-point "送丝"
```

试跑时可以限制记录数：

```powershell
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-video-records 20 `
  --max-document-records 80
```

## LLM 上下文长度与大素材风险

素材目录达到几个 G、几十 G 甚至上百 G 时，不能把整本教材的全部视频转写、PPT 文本、表格文本一次性塞进大模型 prompt。大模型真正接收的是 `02_working_processing/json/` 下 JSONL 中抽取后的文本证据，不会接收视频原文件；但如果某一章挂载的转写和文档证据过多，仍然可能出现 prompt 过长、超过上下文窗口、生成失败或生成质量下降的问题。

当前脚本已经提供几种规避方式：

```powershell
# 按章生成，避免整本一次性进入模型
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --use-llm

# 按知识点/小节生成，适合先做局部验证
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --knowledge-point "送丝" `
  --use-llm

# 试跑时限制记录数量，先验证链路和质量
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-video-records 20 `
  --max-document-records 80 `
  --use-llm

# 限制每个知识点最多使用的证据块数量
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-chunks-per-knowledge-point 8 `
  --use-llm

# 开启自动 token 预算，防止单章证据过多导致 prompt 超长
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-input-tokens 60000 `
  --max-tokens-per-evidence-chunk 1200 `
  --use-llm

# 更稳的生产模式：保留高价值原证据，并把超预算证据分批压缩为摘要证据
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-input-tokens 60000 `
  --max-tokens-per-evidence-chunk 1200 `
  --summarize-over-budget `
  --summary-token-reserve-ratio 0.3 `
  --max-tokens-per-summary-chunk 500 `
  --max-summary-source-chunks 8 `
  --use-llm
```

推荐生产流程：

1. 先用 `--book-mode` 读取 `asset_block_map.xlsx` 生成 `book_plan.json` 和 `book_outline.md`。
2. 流程按章派生章节计划，每章独立应用 token 预算，避免整本教材一次性进入模型。
3. 最后自动汇总为完整 `digital_book.json` 和 `digital_book.zip`。
4. 大素材目录默认不要使用 `--copy-media-assets`，只引用原处理目录中的视频和图片路径。

后续可继续增强的生产级能力：

- 按不同 LLM 模型自动匹配上下文窗口和预算阈值。
- 更细的素材质量打分：按 ASR 质量、视频清晰度、知识点相关度自动调整主素材/引用素材权重。
- 更多素材类型自动接入：自动扫描 `02_working_processing/json/` 下的题库、表格、文档段落等 JSONL，并按策略纳入教材生成。
- 更强的结构去噪：继续过滤课程标准表、教学计划表、文件名、目录名等不适合直接进入学生正文的素材。

当前已经提供 token 预算与分批摘要机制：传入 `--max-input-tokens` 后，流程会先估算进入编排的证据文本长度，超过预算时会压缩单条超长证据，并按章节/知识点轮询保留教学价值、相关性、置信度和审核状态更好的证据。若同时开启 `--summarize-over-budget`，流程会把被挤出的证据按章节/知识点分批摘要为新的证据块，并保留 `source_chunk_ids` 追溯来源。预算结果会写入 `agent_workflow/artifact_manifest.json` 的 summary 中，包括 `token_budget_enabled`、`token_budget_original_estimated_tokens`、`token_budget_kept_estimated_tokens`、`token_budget_kept_source_chunks`、`token_budget_truncated_chunks`、`token_budget_dropped_chunks`、`token_budget_summary_chunks`、`token_budget_summarized_source_chunks` 和 `token_budget_uncovered_dropped_chunks`。

因此，当前流程已经能通过按章、按知识点、限量参数、token 预算和分批摘要规避超长 prompt。后续还可以继续增强为按不同 LLM 模型自动匹配上下文窗口。

## 当前实现状态、不足与注意事项

### 已经实现

当前仓库已经打通“本地素材目录 -> XLSX 全书规划 -> 结构化证据 -> 多智能体教材生成 -> 前端可阅读电子教材”的主流程。只要同伴电脑上存在 `work_materials/work_material1/`，并且其中已经有 `01_manifest_inventory/asset_block_map.xlsx` 与 `02_working_processing/json/video_segments.jsonl`，就可以通过 `scripts/run_full_digital_textbook.py --book-mode` 生成一套整本教材产物。

已经具备的能力：

- 读取本地素材处理结果：默认读取 `video_segments.jsonl`，自动读取 `ppt_assets.jsonl`，也支持通过 `--document-segments` 额外接入表格、题库、考核表等 JSONL。
- 读取同伴整理的 XLSX：`asset_block_map.xlsx` 可直接作为全书规划入口，`material_block_cn` 作为章，`knowledge_point_cn` 作为节/知识点。
- 自动生成课程排序表：流程会输出 `curriculum_order.generated.yml`，记录从 XLSX 和内置焊接课程规则推导出的章节顺序，后续可以人工固定为课程模板，也可以继续自动生成。
- 支持整本教材模式：`--book-mode` 会先生成 `book_plan.json`、`book_outline.md` 和 `book_plan_review.md/json`，再按章派生当前兼容的章节计划并汇总成一本数字教材。
- 统一素材证据结构：视频片段、PPT 页、文档段落、结构化表格等会被转换成统一的 `EvidenceChunk`，后续 agent 不直接面对不同原始格式。
- 支持多种生成粒度：可以整本生成，也可以通过 `--chapter` 按章生成，通过 `--knowledge-point` 按知识点/小节生成。
- 支持大素材目录：默认不复制视频和关键帧到电子教材包，而是在 `digital_book.json` 中引用处理目录里的原路径，避免几个 G 或上百 G 素材被重复复制。
- 生成教材过程产物：会生成大纲、证据索引、章节计划、初稿、审查报告、修订稿、Word 文档、工作流摘要和产物清单。
- 生成前端电子教材：会输出 `digital_book/digital_book.json`、`index.html`、`styles.css`、`app.js` 等文件，可由浏览器打开阅读，并能嵌入视频块；左侧目录支持章/节折叠和锚点跳转。
- 生成一键打开脚本：仓库根目录提供 `open_digital_book.bat`，也可运行 `python scripts/open_digital_book.py` 自动启动本地服务并打开最新正式教材。
- 已接入多智能体编排：包含资源分析、知识组织、标题润色、活动设计、案例设计、正文写作、事实核验、教学审查、修订和电子教材检查。
- 支持可选 LLM：默认可以不使用 LLM 跑通流程；加 `--use-llm` 后，资源分析、写作、事实核验、教学审查和修订等环节可以调用 OpenAI-compatible 大模型。
- 支持 LLM 缓存与重试：开启 LLM 时可使用缓存，减少重复调用；也有基础重试参数。

### 目前不足

当前流程属于已经打通的 MVP/研发闭环，还不是完全生产级自动流水线。主要不足如下：

- XLSX 已经可以参与全书规划，但它主要解决章、节、知识点归属；正文证据仍依赖已转换好的 JSON/JSONL。也就是说，`asset_block_map.xlsx` 不能替代 ASR、PPT 抽取、文档抽取等前置处理结果。
- 原始素材预处理不是本脚本完整负责的范围。视频转码、ASR、关键帧抽取、PPT 图片导出、表格抽取等需要前置处理结果已经存在。
- ASR 纠错还不是完整闭环。现在 LLM 可做资源分析和文本规范化，但还没有专门的“术语表驱动 ASR 纠错 agent -> 自动质量标记 -> 回写转写结果”的完整流程。
- 素材类型自动接入还不够完整。PPT 默认接入，其他 JSONL 类型可以手动传入，但还没有自动扫描 `02_working_processing/json/` 下所有可用素材类型并按策略纳入。
- token 预算与分批摘要已有基础实现。可以通过 `--max-input-tokens` 控制总证据规模，通过 `--summarize-over-budget` 把超预算证据压缩为摘要证据；但还没有做到按不同 LLM 模型自动匹配上下文窗口，也没有独立的摘要质量评分报告来自动发现压缩过度或信息丢失。
- 前端阅读器已经可用，但还不是完整平台级阅读系统。当前重点是展示电子教材、结构化内容和视频块；学习记录、题目作答、进度追踪、教师端审核等还未实现。
- 事实核验和教学深审依赖输入证据质量。若转写乱码、章节归类错误、时间码未复核，审查报告会提示风险，但不能保证自动修复所有源头问题。
- 自动生成的章节顺序可能和教材专家规划不完全一致。当前通过内置焊接课程顺序和 `curriculum_order.generated.yml` 降低漂移风险，但如果学校已有固定课程标准，建议后续把该 YAML 固化为正式课程模板。

### 使用时需要注意

- 大素材目录默认不要加 `--copy-media-assets`。该参数会把媒体复制进 `digital_book/assets/`，只适合小样例包或需要离线分发的小范围教材。
- 用默认大素材模式时，浏览器服务目录应指向 `work_materials/work_material1`，不要只指向 `digital_book` 目录，否则相对视频路径可能无法访问。推荐直接用一键打开脚本：

```powershell
python scripts/open_digital_book.py
```

访问：

```text
http://127.0.0.1:8767/05_final_deliverables/digital_book/index.html
```

- 上百 G 素材不要直接整本无限量开启 `--use-llm` 硬跑。建议使用 `--book-mode`，并按章应用 `--max-chapter-input-tokens`、`--max-video-records`、`--max-document-records`、`--max-chunks-per-knowledge-point` 和 `--max-input-tokens`。
- 若输出内容出现乱码、错章、错知识点，优先检查 `02_working_processing/json/*.jsonl` 中的 `recommended_chapter`、`knowledge_point`、`evidence_text`、`source_video`、`image_paths` 等字段。
- 若视频不能播放，优先检查 `digital_book.json` 中视频 `src` 指向的相对路径，以及本地 HTTP 服务根目录是否正确。
- 当前项目约定不设置人工审核门槛，系统会自动生成可阅读教材；但自动教材仍可能受 ASR 错误、素材错归类、表格噪声和专家顺序差异影响。后续若要面向正式出版或学校平台上线，建议另设抽检或专家定稿流程。
