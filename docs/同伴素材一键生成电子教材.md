# 同伴素材一键生成电子教材

适用场景：素材、台账、切片、PPT 证据和转码结果都在同伴电脑本地的 `work_materials/work_material1/` 目录中；同伴拉取本仓库更新后，希望直接跑出可打开的电子教材。

## 目录要求

脚本默认读取：

```text
work_materials/work_material1/
├── 01_manifest_inventory/
│   ├── assets_manifest.xlsx
│   ├── active_assets.xlsx
│   ├── asset_block_map.xlsx
│   ├── chapter_asset_map_reviewed.xlsx
│   ├── ppt_assets.xlsx
│   └── video_segments.xlsx
└── 02_working_processing/
    ├── converted_mp4/
    ├── keyframes/
    ├── ppt_images/
    └── json/
        ├── video_segments.jsonl
        └── ppt_assets.jsonl
```

其中 xlsx 是人工可读台账，程序优先读取 `02_working_processing/json/` 下的 JSONL。

## 一键生成

在仓库根目录运行：

```powershell
pip install -r requirements.txt
python scripts/run_full_digital_textbook.py --material-root work_materials/work_material1 --title "钨极氩弧焊数字教材"
```

默认不会复制视频大文件，而是在 `digital_book.json` 中用相对路径引用 `02_working_processing/converted_mp4/` 和 `02_working_processing/keyframes/`。这适合几个 G 到上百 G 的素材目录，避免每次生成教材都重复复制媒体资源。

生成结果：

```text
work_materials/work_material1/05_final_deliverables/
├── agent_workflow/
│   ├── textbook_outline.md/json
│   ├── evidence_chunks.jsonl
│   ├── chapter_plan.json
│   ├── textbook_draft.md/docx
│   ├── review_report.md/json
│   ├── textbook_final.md/docx
│   ├── workflow_summary.json
│   └── artifact_manifest.json
└── digital_book/
    ├── digital_book.json
    ├── index.html
    ├── styles.css
    ├── app.js
    ├── ask_config.js
    └── assets/
```

打开电子教材：

```powershell
python -m http.server 8110 --bind 127.0.0.1 --directory work_materials/work_material1
```

浏览器访问：

```text
http://127.0.0.1:8110/05_final_deliverables/digital_book/
```

如果需要生成一个可拷贝给别人、媒体资源也打包进 `digital_book/assets/` 的小样例包，可以显式复制媒体：

```powershell
python scripts/run_full_digital_textbook.py --material-root work_materials/work_material1 --copy-media-assets
```

注意：素材达到几十 G 或上百 G 时，不建议使用 `--copy-media-assets`。

## 分章/分节生成

大教材建议按章或按知识点分批生成，避免一次处理全量证据和全量媒体：

```powershell
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作"
```

只生成某个知识点：

```powershell
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --knowledge-point "送丝"
```

试跑时可以限制记录数：

```powershell
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-video-records 20 `
  --max-document-records 80
```

## 可选 LLM

如果要让资源分析、正文生成、事实核验、教学深审和修订使用 OpenAI-compatible LLM：

```powershell
copy .env.example .env
# 填入 ECNU_PLUS_API_KEY / ECNU_PLUS_BASE_URL / ECNU_PLUS_MODEL

```

## LLM 上下文长度与大素材风险

素材目录达到几个 G、几十 G 甚至上百 G 时，不能把整本教材的全部视频转写、PPT 文本、表格文本一次性塞进大模型 prompt。大模型真正接收的是 `02_working_processing/json/` 下 JSONL 中抽取后的文本证据，不会接收视频原文件；但如果某一章挂载的转写和文档证据过多，仍然可能出现 prompt 过长、超过上下文窗口、生成失败或生成质量下降的问题。

当前脚本已经提供几种规避方式：

```powershell
# 按章生成，避免整本一次性进入模型
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --use-llm

# 按知识点/小节生成，适合先做局部验证
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --knowledge-point "送丝" `
  --use-llm

# 试跑时限制记录数量，先验证链路和质量
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-video-records 20 `
  --max-document-records 80 `
  --use-llm

# 限制每个知识点最多使用的证据块数量
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-chunks-per-knowledge-point 8 `
  --use-llm

# 开启自动 token 预算，防止单章证据过多导致 prompt 超长
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-input-tokens 60000 `
  --max-tokens-per-evidence-chunk 1200 `
  --use-llm

# 更稳的生产模式：保留高价值原证据，并把超预算证据分批压缩为摘要证据
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --chapter "钨极氩弧焊基本操作" `
  --max-input-tokens 60000 `
  --max-tokens-per-evidence-chunk 1200 `
  --summarize-over-budget `
  --summary-token-reserve-ratio 0.3 `
  --max-tokens-per-summary-chunk 500 `
  --max-summary-source-chunks 8 `
  --use-llm
```

推荐生产流程：

1. 先按知识点小样本试跑，确认素材路径、视频播放、章节归类和 LLM 输出质量正常。
2. 再按章生成，每章控制输入证据规模。
3. 最后把各章结果合并成整本电子教材。
4. 大素材目录默认不要使用 `--copy-media-assets`，只引用原处理目录中的视频和图片路径。

后续需要补充的生产级能力：

- 自动 token 预算器：每次调用 LLM 前估算证据文本长度，超过阈值就自动裁剪、筛选或摘要压缩。
- 证据预筛选：按章节、知识点、教学价值、相关性、置信度和审核状态优先选择高价值素材。
- 分批摘要再合成：开启 `--summarize-over-budget` 后，会为摘要预留一部分 token，把超预算证据按章节/知识点分批合成为 `summary_evidence`，再交给写作、事实核验和教学审查 agent。
- 批量分章任务队列：读取章节清单后自动逐章生成，避免人工反复输入命令。

当前已经提供 token 预算与分批摘要机制：传入 `--max-input-tokens` 后，流程会先估算进入编排的证据文本长度，超过预算时会压缩单条超长证据，并按章节/知识点轮询保留教学价值、相关性、置信度和审核状态更好的证据。若同时开启 `--summarize-over-budget`，流程会把被挤出的证据按章节/知识点分批摘要为新的证据块，并保留 `source_chunk_ids` 追溯来源。预算结果会写入 `agent_workflow/artifact_manifest.json` 的 summary 中，包括 `token_budget_enabled`、`token_budget_original_estimated_tokens`、`token_budget_kept_estimated_tokens`、`token_budget_kept_source_chunks`、`token_budget_truncated_chunks`、`token_budget_dropped_chunks`、`token_budget_summary_chunks`、`token_budget_summarized_source_chunks` 和 `token_budget_uncovered_dropped_chunks`。

因此，当前流程已经能通过按章、按知识点、限量参数、token 预算和分批摘要规避超长 prompt。后续还可以继续增强为按不同 LLM 模型自动匹配上下文窗口。

## 当前实现状态、不足与注意事项

### 已经实现

当前仓库已经打通“本地素材目录 -> 结构化证据 -> 多智能体教材生成 -> 前端可阅读电子教材”的主流程。只要同伴电脑上存在 `work_materials/work_material1/`，并且其中已经有 `02_working_processing/json/video_segments.jsonl`，就可以通过 `scripts/run_full_digital_textbook.py` 生成一套教材产物。

已经具备的能力：

- 读取本地素材处理结果：默认读取 `video_segments.jsonl`，自动读取 `ppt_assets.jsonl`，也支持通过 `--document-segments` 额外接入表格、题库、考核表等 JSONL。
- 统一素材证据结构：视频片段、PPT 页、文档段落、结构化表格等会被转换成统一的 `EvidenceChunk`，后续 agent 不直接面对不同原始格式。
- 支持多种生成粒度：可以整本生成，也可以通过 `--chapter` 按章生成，通过 `--knowledge-point` 按知识点/小节生成。
- 支持大素材目录：默认不复制视频和关键帧到电子教材包，而是在 `digital_book.json` 中引用处理目录里的原路径，避免几个 G 或上百 G 素材被重复复制。
- 生成教材过程产物：会生成大纲、证据索引、章节计划、初稿、审查报告、修订稿、Word 文档、工作流摘要和产物清单。
- 生成前端电子教材：会输出 `digital_book/digital_book.json`、`index.html`、`styles.css`、`app.js` 等文件，可由浏览器打开阅读，并能嵌入视频块。
- 已接入多智能体编排：包含资源分析、知识组织、标题润色、活动设计、案例设计、正文写作、事实核验、教学审查、修订和电子教材检查。
- 支持可选 LLM：默认可以不使用 LLM 跑通流程；加 `--use-llm` 后，资源分析、写作、事实核验、教学审查和修订等环节可以调用 OpenAI-compatible 大模型。
- 支持 LLM 缓存与重试：开启 LLM 时可使用缓存，减少重复调用；也有基础重试参数。

### 目前不足

当前流程属于已经打通的 MVP/研发闭环，还不是完全生产级自动流水线。主要不足如下：

- xlsx 目前仍主要作为人工台账，生成主流程优先读取已经转换好的 JSON/JSONL；还没有把“从 xlsx 自动识别所有素材源并生成统一输入清单”完全纳入一键脚本。
- 原始素材预处理不是本脚本完整负责的范围。视频转码、ASR、关键帧抽取、PPT 图片导出、表格抽取等需要前置处理结果已经存在。
- ASR 纠错还不是完整闭环。现在 LLM 可做资源分析和文本规范化，但还没有专门的“术语表驱动 ASR 纠错 agent -> 人工复核 -> 回写转写结果”的完整流程。
- 素材类型自动接入还不够完整。PPT 默认接入，其他 JSONL 类型可以手动传入，但还没有自动扫描 `02_working_processing/json/` 下所有可用素材类型并按策略纳入。
- token 预算与分批摘要已有基础实现。可以通过 `--max-input-tokens` 控制总证据规模，通过 `--summarize-over-budget` 把超预算证据压缩为摘要证据；但还没有做到按不同 LLM 模型自动匹配上下文窗口，也没有独立的人工复核界面来查看每个摘要证据的压缩质量。
- 批量分章生成还没有完全自动化。当前可以手动按章/知识点跑，后续需要支持读取章节清单后自动逐章生成，并最终合并整本。
- 前端阅读器已经可用，但还不是完整平台级阅读系统。当前重点是展示电子教材、结构化内容和视频块；学习记录、题目作答、进度追踪、教师端审核等还未实现。
- 事实核验和教学深审依赖输入证据质量。若转写乱码、章节归类错误、时间码未复核，审查报告会提示风险，但不能保证自动修复所有源头问题。

### 使用时需要注意

- 大素材目录默认不要加 `--copy-media-assets`。该参数会把媒体复制进 `digital_book/assets/`，只适合小样例包或需要离线分发的小范围教材。
- 用默认大素材模式时，浏览器服务目录应指向 `work_materials/work_material1`，不要只指向 `digital_book` 目录，否则相对视频路径可能无法访问。

```powershell
python -m http.server 8110 --bind 127.0.0.1 --directory work_materials/work_material1
```

访问：

```text
http://127.0.0.1:8110/05_final_deliverables/digital_book/
```

- 上百 G 素材不要直接整本开启 `--use-llm` 硬跑。建议先按知识点试跑，再按章生成，必要时加 `--max-video-records`、`--max-document-records`、`--max-chunks-per-knowledge-point` 和 `--max-input-tokens`。
- 若输出内容出现乱码、错章、错知识点，优先检查 `02_working_processing/json/*.jsonl` 中的 `recommended_chapter`、`knowledge_point`、`evidence_text`、`source_video`、`image_paths` 等字段。
- 若视频不能播放，优先检查 `digital_book.json` 中视频 `src` 指向的相对路径，以及本地 HTTP 服务根目录是否正确。
- 正式交付前需要人工抽检：视频时间码、ASR 文本、PPT 证据、事实表述、教学活动和案例题答案都应抽样复核。
- 当前建议把本流程当成“生成教材初稿和可阅读电子教材包”的自动化工具，而不是无人值守的最终出版系统。
