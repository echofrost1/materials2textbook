# 多智能体编排开发计划

> 适用分工：同伴继续按照 `material_pipeline_forward_plan.md` 做资料台账、去重、分类、素材大块和切片处理；本计划负责承接这些中间产物，完成知识组织、教材生成、审核修订的多智能体编排。

---

## 1. 分工边界

### 上游数据处理线负责

- 原始资料只读保存。
- 生成资料台账、去重表、有效素材表。
- 建立素材大块和多归属映射。
- 对选中素材做转码、ASR、关键帧、人工时间码。
- 输出 `video_segments.xlsx` 和 `video_segments.jsonl`。

### 多智能体编排线负责

- 把 `video_segments.jsonl` 转换为统一 `EvidenceChunk`。
- 按章节和知识点组织证据。
- 生成 `ChapterPlan`。
- 生成教材草稿。
- 执行事实审核、引用审核、教学审核。
- 输出修订版教材和审核报告。

两条线的接口是：

```text
work_material1/02_working_processing/json/video_segments.jsonl
        ↓
evidence_chunks.jsonl
        ↓
chapter_plan.json
        ↓
textbook_draft.md
        ↓
review_report.json
        ↓
textbook_final.md
```

## 2. MVP 目标

第一版不追求大模型完全自动生成高质量教材，而是先跑通可复核闭环：

```text
视频片段 JSONL
→ 标准证据片段
→ 章节计划
→ 教材草稿
→ 审核报告
→ 修订提示版教材
```

MVP 验收标准：

- 能读取上游 `video_segments.jsonl`。
- 每个片段能转换为 `EvidenceChunk`。
- 能按 `recommended_chapter` 和 `knowledge_point` 生成章节计划。
- 教材草稿中每个知识点都有证据引用。
- 审核报告能指出待人工复核、缺少证据、教学结构不足等问题。
- 输出文件能被人工直接打开检查。

### 3.1 Agent 角色

| Agent | 当前实现 | 后续增强 |
|---|---|---|
| ResourceAnalystAgent | 规则转换 `video_segments` 到 `EvidenceChunk`，`--use-llm` 时可保守规范化摘要、关键词和证据文本；LLM 调用默认写入持久化缓存，并通过 `RetryingLLMProvider` 进行失败重试 | 后续继续优化批量调度 |
| KnowledgeOrganizerAgent | 按章节和知识点分组，并输出 `learning_path`、`difficulty_level`、`cluster_id` 和 `prerequisite_ids`；支持显式顺序字段、上游语义聚类元数据和规则化先修关系 | 后续接入更精细的章节顺序优化 |
| TitlePolisherAgent | 基于本章素材大块和知识点难度，保守润色章节、目录和知识点标题，不改变证据范围、学习路径或 `chunk_id` 绑定 | 后续继续接入 LLM 风格统一和人工可编辑标题表 |
| TextbookWriterAgent | 生成带证据引用、学习路径、结构化活动和案例示例的 Markdown 草稿；`--use-llm` 时可由 LLM 生成正文 | 后续继续增强更自然的案例叙事 |
| CaseDesignerAgent | 基于章节计划和证据片段生成带学生实训情境、迁移判断、例题、参考分析和证据绑定的案例示例，待复核片段会保留谨慎表述 | 后续继续细化真实项目案例库 |
| EvidenceReviewerAgent | 检查证据缺失、待审核、评分不足，并审核正文是否保留有效 `chunk_id` 引用；支持段落级和断言级事实支撑评分，识别无引用断言、未知引用、待复核证据未标注和同一证据/主题下的要求性-禁止性冲突；`--use-llm` 时追加证据支撑和引用覆盖深审；输出证据覆盖率、引用覆盖率、段落事实支撑率、断言事实支撑率、可正式使用证据率 | 后续继续细化更复杂的跨段落语义一致性 |
| PedagogyReviewerAgent | 检查学习目标、知识点数量、学习路径、结构化活动、练习证据绑定、评价量规、难度梯度、学生情境案例和迁移判断；`--use-llm` 时追加教学目标、难度梯度和活动支撑深审；输出教学结构完整度、活动质量评分和案例质量评分 | 后续继续细化学生画像适配细分规则 |
| RevisionAgent | 规则模式汇总审核建议到教材末尾；`--use-llm` 时按审核意见重写完整正文；输出 `revision_diff.md` 展示草稿到最终稿差异和人工确认清单 | 后续继续细化人工确认状态回写 |

当前工作流还会额外生成：

- `textbook_outline.md/json`：严格由上游素材片段生成的三级教材目录，当前规则为“章 = recommended_chapter，节 = material_block，知识点 = knowledge_point”。
- `evidence_index.md`：按章节和知识点组织的人工可读证据索引，便于教师检查来源、时间码、状态和关键帧。
- `workflow_summary.json`：统计来源记录数、进入编排片段数、章节数、知识点数、审核问题数量和片段状态分布。
- `review_report.md`：给人工审核看的 Markdown 版报告，比 JSON 更适合开会和分工。
- `review_history.json`：记录多轮审核-修订闭环中每一轮的问题数量和审核报告。
- `revision_diff.md`：记录草稿到最终稿的 Markdown 差异，并把审核建议转成教师确认 checklist。

## 4. 运行方式

使用当前样例数据运行：

```powershell
python scripts/run_pipeline.py
```

如果只想运行编排、不运行前置校验：

```powershell
python scripts/run_agent_workflow.py
```

指定输入输出：

```powershell
python scripts/run_agent_workflow.py `
  --segments work_material1/02_working_processing/json/video_segments.jsonl `
  --output-dir work_material1/05_final_deliverables/agent_workflow `
  --title "钨极氩弧焊数字教材样章"
```

输出：

```text
work_material1/05_final_deliverables/agent_workflow/
├── textbook_outline.md
├── textbook_outline.json
├── evidence_chunks.jsonl
├── evidence_index.md
├── chapter_plan.json
├── textbook_draft.md
├── textbook_draft.docx

├── review_report.md
├── revision_diff.md
├── workflow_summary.json
├── artifact_manifest.json
├── textbook_final.md
└── textbook_final.docx
```

常用运行模式：

```powershell
# 草稿模式：允许待人工复核片段进入教材草稿，但会在审核报告中标注
python scripts/run_agent_workflow.py

# 正式模式：只使用 review_status 包含 approved 的片段
python scripts/run_agent_workflow.py --approved-only

# 控制每个知识点最多引用几个证据片段
python scripts/run_agent_workflow.py --max-chunks-per-knowledge-point 2

# 过滤教学价值评分过低的片段
python scripts/run_agent_workflow.py --min-teaching-value 0.6

# 多轮审核-修订闭环：第 1 轮审核后修订，再进入第 2 轮审核
python scripts/run_agent_workflow.py --review-rounds 2
```

## 5. 近期开发计划

### 阶段 1：接口稳定

- 固定 `EvidenceChunk`、`ChapterPlan`、`ReviewReport` 数据结构。
- 和上游确认 `video_segments.jsonl` 字段不频繁变动。
- 对 `Pending_Manual_Timecode`、`approved`、`rejected` 等状态建立过滤规则。
- 使用 `scripts/validate_video_segments.py` 校验上游片段字段、时间码、重复 ID、转写状态和证据完整性。

### 阶段 2：样章生成

- 基于 TIG 样例生成一章教材草稿。
- 严格基于上游片段生成三级教材目录，不主动补充素材中没有的章节。
- 每个段落保留 `chunk_id`、原视频、时间码。
- 标出 ASR 质量低或待人工复核的片段。

### 阶段 3：审核闭环

- 事实审核：检查正文结论是否有证据片段支撑。
- 引用审核：检查每个引用能否回到原始素材和时间码。
- 正文引用审核：检查教材草稿是否丢失计划中的证据 `chunk_id`，以及是否引用了不存在的证据。
- 教学审核：检查章节目标、知识点顺序、活动设计。

### 阶段 4：接入真实 LLM

- 已抽象 OpenAI-compatible Provider，可通过 `ECNU_PLUS_API_KEY`、`ECNU_PLUS_BASE_URL`、`ECNU_PLUS_MODEL` 接入 `ecnu-plus`。
- 已支持 `RetryingLLMProvider`，`--use-llm` 时默认对失败调用重试 2 次，可用 `--llm-max-retries` 和 `--llm-retry-backoff` 调整。
- 已支持 `ResourceAnalystAgent` 在 `--use-llm` 模式下保守规范化片段摘要、关键词和证据文本，并保留原始证据文本到 metadata。
- 已支持 `python scripts/run_agent_workflow.py --use-llm` 用 LLM 替换 TextbookWriterAgent 的正文生成。
- 已支持 `RevisionAgent` 在 `--use-llm` 模式下根据审核报告生成修订后的完整 Markdown。
- 已支持 `EvidenceReviewerAgent` 在 `--use-llm` 模式下追加事实支撑、引用覆盖和待复核表述深审。
- 已支持 `PedagogyReviewerAgent` 在 `--use-llm` 模式下追加教学目标、难度梯度和活动支撑深审。
- 已支持 `--review-rounds` 执行多轮审核-修订闭环，并输出 `review_history.json`。
- 已支持在 `workflow_summary.json`、`review_report.md` 和 `artifact_manifest.json` 输出证据覆盖率、引用覆盖率、段落事实支撑率、可正式使用证据率、教学结构完整度和综合质量评分。
- 已支持 `TitlePolisherAgent` 对章节、目录和知识点标题做保守润色，例如将泛标题与素材大块结合为更适合教材阅读的标题，同时保留原有证据绑定。
- 已支持 `ActivityDesignerAgent` 生成观察定位、复述解释、分析迁移三层结构化活动，并绑定知识点、证据片段和评价量规。
- 已支持 `CaseDesignerAgent` 生成基于证据片段的案例示例，输出学生实训情境、迁移判断、例题、参考分析和 `evidence_chunk_ids`，并进入 Markdown 草稿和数字教材 `case_example` 块。
- 已支持 `PedagogyReviewerAgent` 审核案例示例是否绑定本章证据、覆盖知识点、具备学生画像/学习情境和迁移应用要求，并在 `workflow_summary.json`、`review_report.md` 和 `artifact_manifest.json` 输出案例质量评分。
- 已支持 `PedagogyReviewerAgent` 对练习质量、证据绑定、评价量规和难度梯度进行规则评分，并在 `workflow_summary.json`、`review_report.md` 和 `artifact_manifest.json` 输出活动质量评分。
- 已支持段落级事实支撑评分：对教材正文块抽取 `chunk_id`，区分已支撑、无引用、未知引用和待复核证据引用，并由 `EvidenceReviewerAgent` 写入审核问题。
- 已支持断言级事实支撑评分：对事实句子抽取/继承 `chunk_id`，区分已支撑、无引用、未知引用和待复核证据引用，并输出断言事实支撑率。
- 已支持轻量跨断言一致性检查：同一证据/主题下如果同时出现“需要/必须”和“不能/禁止”等冲突表述，会被 `EvidenceReviewerAgent` 标为高风险问题。
- 后续继续细化更复杂的跨段落语义一致性和学生画像适配细分规则。

### 阶段 5：扩展资料类型

- 已支持通过 `--document-segments` 接入上游已解析的 PDF / PPT / Markdown / Text 片段。
- 已新增 `document_segment_to_evidence_chunk`，将文档段落、页码、标题层级、来源路径和教学评分统一转为 `EvidenceChunk`。
- 已支持在章节规划时同时使用视频证据和文档证据。
- 当前边界：编排线接收 `document_segments.jsonl`，不直接解析二进制 PDF/PPT；二进制解析仍属于上游资料处理线。

新增 Agent / Exporter 规划：

| 模块 | 职责 | 第一版实现 |
|---|---|---|
| MediaBinderAgent | 把视频、关键帧、图片绑定到对应任务和知识点 | 第一版由 `DigitalBookExporter` 根据 `EvidenceChunk.locator`、`asset_id` 和 `keyframe_paths` 规则绑定 |

前端阅读器第一版能力：

- 左侧目录：项目、任务、情境导入、任务实施、思考与练习。
- 正文阅读区：图文混排，保留知识点和重点词。
- 视频播放：使用 HTML5 `<video controls>`，支持封面图和片段起止时间。
- 资源追溯：学生端不展示 `chunk_id`、原视频、时间码和审核状态；教师端仍可从 `metadata.teacher_evidence` 追溯每个正文块对应的素材来源。
- 学习活动：展示任务评价和思考题。
- 阅读辅助：已支持全文搜索、阅读进度、字号缩放、当前任务书签、本地学习笔记、本地检索式 AI 问书，以及学习数据 JSON 包导出/导入；已支持命令行问书脚本，默认检索教材片段并列证据来源，`--use-llm` 时可基于检索片段生成回答；阅读器已输出 `ask_config.js`，可配置教师端 `/ask` 服务，把页面命中的教材片段交给服务端 LLM 生成证据约束回答，服务不可用时自动回退本地检索；已支持配置学习数据同步端点，把阅读进度、笔记和书签 POST 到教师端收集目录；已支持离线聚合多个学习数据包生成班级学习报告，并输出可直接打开的 HTML 可视化页面。

- 能打开 `digital_book/index.html` 看到完整电子教材。
- 左侧目录能跳转到对应项目和任务。
- 正文包含“情境导入、任务实施、思考与练习”等项目化教材结构。
- 视频能在页面中播放，并显示关键帧封面。
- 每个视频和知识点能追溯到 `chunk_id`、原素材和时间码。
- 能从当前 TIG 样例自动生成 `digital_book.json`。

完成状态审计见：`docs/多智能体编排完成审计.md`。

## 7. 学生版数字教材发布包

学生可阅读的 `digital_book.zip` 必须使用专门的学生包脚本生成：

```powershell
python scripts/package_digital_book.py `
  work_materials/work_material1/05_final_deliverables_chapter_quality_test/digital_book `
  --output digital_book.zip `
  --asset-fallback-zip digital_book.zip `
  --ask-smoke-question "送丝操作要注意什么" `
  --ask-smoke-expected "送丝操作要点" `
  --ask-smoke-expected "送丝 视频片段"
```

也可以在完整样章生成命令中直接输出学生包：

```powershell
python scripts/run_full_digital_textbook.py `
  --material-root work_materials/work_material1 `
  --title "整章教材化复检" `
  --chapter "钨极氩弧焊基本操作" `
  --max-video-records 80 `
  --max-document-records 300 `
  --output-dir work_materials/work_material1/05_final_deliverables_chapter_quality_test/agent_workflow `
  --student-package-output digital_book.zip `
  --student-package-asset-fallback-zip digital_book.zip `
  --student-package-ask-smoke-question "送丝操作要注意什么" `
  --student-package-ask-smoke-expected "送丝操作要点" `
  --student-package-ask-smoke-expected "送丝 视频片段"
```

如果已经拿到了一个 `digital_book.zip`，只做发布前验收、不重新打包：

```powershell
python scripts/validate_digital_book_package.py digital_book.zip `
  --ask-smoke-question "送丝操作要注意什么" `
  --ask-smoke-expected "送丝操作要点" `
  --ask-smoke-expected "送丝 视频片段" `
  --max-package-mb 2048
```

发布包规则：

- 只包含阅读器运行所需文件：`index.html`、`app.js`、`styles.css`、`ask_config.js`、`digital_book.json` 和 `assets/`。
- 不包含 `digital_book_review.json`、`digital_book_review.md` 等审核报告。
- `digital_book.json` 中会移除 `evidence_chunk_ids`、`teacher_evidence`、`chunk_id` 等教师追溯字段。
- 学生前端仍可阅读正文、播放视频、使用本地 AI 问书；教师侧追溯材料保留在 workflow 输出目录和审核报告中，不进入学生发布包。
- `scripts/package_digital_book.py` 默认会运行学生包校验；如果缺少必要文件、缺少媒体资产、混入 review 文件、混入内部追溯字段，命令会以非 0 状态退出。
- 可选的问书 smoke test 会校验指定问题能命中预期正文/视频结果，避免只通过字段清理但问书仍返回“未找到”或混入学习路径。
- 如果学生包校验失败，`package_digital_book.py` 和 `run_full_digital_textbook.py --student-package-output ...` 会删除刚生成的无效 zip，避免误发失败产物。
- 学生包输出采用临时 zip 校验后原子替换；如果覆盖已有 `digital_book.zip` 时新包校验失败，旧包会保留，不会被坏包覆盖或删除。
- 继承旧包媒体资产时会拒绝 `..`、绝对路径、Windows 反斜杠/盘符等不安全 zip 成员；独立校验脚本也会报告这类 unsafe path。
- 学生包默认体积上限为 2048 MB，可通过 `--max-package-mb` 或 `--student-package-max-mb` 调整；设为 `0` 可关闭。也可通过 `--max-asset-files` / `--student-package-max-asset-files` 限制媒体文件数量。
- 打包时会把视频和关键帧引用重写为包内 `assets/...` 路径；校验器会拒绝 `../../02_working_processing/...` 等外部媒体引用，避免 zip 里有视频但前端离线播放不到。
- 发布脚本默认执行静态 smoke test，检查 `index.html` 引用的 `app.js`、`styles.css`、`ask_config.js`、`digital_book.json` 以及教材 JSON 中的视频/关键帧引用都能在 zip 内找到；必要时可用 `--skip-static-smoke` 或 `--student-package-skip-static-smoke` 跳过。
